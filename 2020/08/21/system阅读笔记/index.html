Managing Google’s data lake: an overview of the GOODS system

一、背景

一个团队设计GOODS（dataset管理工具）的故事，这个GOODS提炼了数据集的元数据和数据集间的关系，为工程师和数据科学家提供支持，而且其使用的是后处理的方式，相当于场外处理，不会影响到现有业务的运行。其挑战的来源是数据的大规模和数据集之间的异构性造成的。
功能包括：search、monitoring services、 DataSet profile、 Annotation services

  • Search: 公司所有数据集的搜索引擎，具有缩小搜索结果的方面，帮助工程师和分析师找到最相关的数据集。
  • DataSet profile: 每个数据集配置文件页面，呈现货物记录的关于特定数据集的元数据，从而帮助用户了解数据集及其与公司其他数据集的关系。profile页面还集成了其他可以进一步处理数据集的工具，从而帮助用户对数据进行操作。
  • Monitoring services: 一种监视服务，允许团队监视其拥有的数据集的特性，如大小、内容中值的分布或可用性。用户可以将此监视服务配置为在功能意外更改时发出警报。
  • Annotation services: 一种注释服务，允许数据集所有者或受信任的主体（例如，数据库管理员或数据管理团队）使用可以出现在配置文件页面中的特定于域的注释来扩展数据集的元数据。例如，数据集所有者可以为数据集的内容提供文本描述，或者附加一个可以通知其他用户的可视化。

二、需要解决的问题

1. 管理数据湖

通过收集数据集的元数据揭示数据集之间的关系。这里可以提出两个问题：
第一点，一个数据集的影响范围有多大。比如一个数据集的产生需要通知哪些团队，回答这个问题需要知道各个团队的组成，数据集所有权和出处关系。
第二点，再比如查找某个代码中使用方法X的特定版本的生产作业的所有数据，回答这个问题需要知道出处、生产作业中二进制文件的版本以及源代码与二进制文件的联系。

2. 元数据提取的可拓展性

这个挑战是GOODS管理的数据集的绝对数量、数据集的每日混乱以及为每个单独的数据集提取元数据的成本造成的。
这里提出了一个方法，是通过数据集的优先级来优先处理重要的数据，但是找到一个合理的度量有点不容易。

3. 后处理的管理方式

在这些数据集被访问或更新后，系统将通过分析来自Google基础设施的信号（例如，通过处理日志或爬网存储系统目录）来收集有关数据集的元数据。而传统EDM是采用api的方式访问数据，势必会占用现有系统的性能。但是这种方法的缺点是，提取和推理数据集元数据变得更加困难，一方面，元数据的不确定性，对内容的分析可能会产生多个候选项，这里需要将所有的候选项记录下来。另一方面，数据集的异构性将导致数据湖中出现各种不同文件类型的元数据，数据湖管理系统必须能够跨各种源存储系统提取和处理元数据，并将这种异构元数据记录在单个catalog中。然而，为不同的元数据使用一个catalog也是在数据湖中部分改善这种异构性的一个机会：catalog可以定义收集到的元数据的一个子集，这些元数据通常在不同类型的数据集中是通用的。这种通用元数据可以为用户和服务提供数据湖中数据集的统一视图。

三、数据湖的关系图

知识图中的节点表示世界上的实体（例如，Tom Hanks，“Forrest Gump”），并且边将这些实体相互连接（例如，Tom Hanks在“Forrest Gump”中扮演了一个角色）。
首先，在数据集之间建立关系（例如，数据集A是数据集B的新版本）。
其次，将数据集链接到其他实体，例如生成数据集的作业、拥有数据集和这些作业的项目和用户，或者定义这些作业的源代码。
可以根据在企业内部收集的各种信息，以事后的方式自动推断出所有这些关系。

GOODS的catalog中的关系

  • 数据集间包含
  • 出处：对于每个数据集，维护数据集如何产生、如何使用、该数据集依赖于哪些数据集以及哪些其他数据集依赖于此数据集的出处。
  • 逻辑集群：同一版本定期生成的数据集；主从复制的数据集；分片的数据集。
  • 内容相似：依靠指纹分析器（字段校验或者哈希值LSH）来确定哪些数据集是彼此的副本，哪些具有不同的内容。

除了将商品目录中的数据集相互链接的关系之外，还依赖于Google的其他基础设施来丰富这个关系图。当系统继续增长时，还有一些当前提取的关系：
•关于产生或读取数据集的工作负责人的信息；
•关于确定数据集可见性权限的数据集所有者和用户组的信息；
•为数据集推断的模式与其在源代码存储库中的定义之间的链接。

关系图可以为新增数据集快速添加关系，同时便于用户实现丰富的查询，并链接到数据湖中的其他数据。

四、系统设计

1. 利用数据湖关系图

首先，正如上文所提，数据湖关系图的出现使得集群可以以损失精度的代价换取提取元数据的成本的降低。因为元数据的规模和异构性，如果提取每个数据集的元数据，系统的负担太大，而且很多数据集是有相似的模式的，所以损失这些精度是值得的。
其次，当提取某个数据集的元数据成本过高或者没有元数据，可以通过提取关系图中其他数据集（eg. 不同版本的）来实现提取，当然这样也会导致数据的不确定性–哪个版本才合适呢？只能把所有候选项都记录下来。
最后，知识图谱的链接可以让GOODS成为探索其他资源的数据的起点。

2. 事后处理模块的协调

利用爬虫或者分析器在各个模块中收集数据，而各个GOODS模块中有着不同的特性，这里就需要去协调各个模块：

首先，不同模块的优先级不同，并不是所有模块的信息都是对时间敏感的，换言之，有些信息是不需要分配太多资源去实时更新的。这里的处理是显式的标识一些关键模块，而一些非关键模块被标识为有空闲资源的时候再去更新。

第二，不同模块间存在需要同步的情况。使用从属模块进行分组，将它们分配在同一个作业中，使从属模块能够检查它们所依赖的所有模块的状态，并作出明智的决定。虽然所有商品模块都将每个数据集的执行状态存储在目录本身中，以避免失败时以及同一作业的不同实例之间的重复工作，但分组相关模块会减少从持久存储后端读取的总字节数。

第三，不同模块的稳定性不同。这里主要是对一些容易崩溃的模块进行隔离，防止它们的崩溃或者长时间挂起影响整个系统。这里的解决方案是，首先用单独的进程去处理高危模块，然后使用一个watchdog线程在某个模块长时间循环的时候将其转为崩溃，防止其一直吃CPU资源，允许pipeline的其他模块运行。

最后，不同模块的时间复杂度不一样，需要不同的计算资源。尽量避免重复运行同一个高复杂度的模块，除非有明确的信号表示信息发生了变化。例如，除非修改数据集，否则内容的指纹不太可能更改，因此指纹分析器可以绕过此类数据集。

3. 搜索排名

通过搜索找到数据集。一个重要的设计是显示给用户在一个逻辑集群里的结果。这样可以以一种有意义的方式压缩搜索结果，而不是用匹配同一个查询的许多相似的数据集压倒用户（例如，显示只在表示其版本的路径的一个组件上不同的数据集）。这里就遇到一个挑战了–排名函数。经过大量的实验（和一些错误的开始），GOODS最终使用了标准的IR-type信号（例如，搜索项与元数据的匹配程度）与从每个数据集的元数据中派生的特定类型信号的综合。
