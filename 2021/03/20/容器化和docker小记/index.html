    容器和docker小记
    2021-03-20
    字数总计: 910|阅读时长: 2 分钟

    学会容器，部署起来，一劳永逸。
    先开个坑…这周的Tut跟了一趟Get Start(https://docs.docker.com/get-started/)

    虚拟化技术让不同操作系统运行在同一台物理环境上，解决了操作系统的异构性；通过池化技术虚拟化可以对硬件资源的有效利用，比如CPU使用率；还有一个是对系统的隔离，虚拟机们使用同一台物理机上对硬件资源的同时，不牺牲安全性。
    容器化是操作系统层面的虚拟化，让同一个操作系统中同时跑多个相互隔离并且共享系统资源的进程。对比起来，容器更轻量级，虚拟化分配的硬件会一直占用着即使一个进程也不跑，也因为资源的分配，虚拟机启动会比容器更久，但是虚拟化的隔离性更好，而容器只能通过内核特性实现一定程度的隔离。

    容器化是怎么做到的？

    命名空间 namespace

    1. Mount： 设定不同容器的根目录（工作目录）
    2. 进程ID：通过进程ID的映射，可以让多个容器内的进程号ID重复
    3. UserID：同样是ID的映射，容器内的用户可以灵活的映射外部的UID，起到权限的控制
    4. 网络端口号映射等…

    控制组 CGROUP

    控制进程对内核内资源CPU、内存、IO的访问权限，通过层级结构来组织的：
    (控制组相关文档：https://docs.docker.com/engine/install/linux-postinstall/)

    Docker

    为啥大家都用Docker?

    依赖管理和到处部署。Docker承诺能帮大家解决依赖冲突、版本、依赖丢失等问题，它的运行环境还允许在各个异构平台都能跑。怎么跟Maven和Java的成功有点那么点相似的意思。

    Docker内的名词

    容器：跑应用的相互隔离的运行环境。
    镜像：只读的build和run应用的模版，就是源代码，Dockerfile（定义镜像的文件，里面写要怎么build，怎么运行应用）这些东西，只读。可以本地存，也能传到中央仓库去。
    数据卷：跑的应用存数据的地方，不配置的话，应用一停，持久化的数据就没。
    网络：做端口映射

    镜像分层

    CopyOnWrite: 所有底层文件都是不可写的，底层的文件可以被多个上层文件复用，如果说某个镜像需要做修改，就在上一层复制要修改的那部分来修改。也因为所有底层文件都是不可写的，启动时候会创多一层文件，做临时存储，在容器结束的时候会删掉这一层，所以应用数据的持久化需要额外处理。
    (DockerFile相关文档：https://docs.docker.com/develop/develop-images/dockerfile_best-practices/)

    数据卷 Data Volume

    1. Bind mount: 早期版本用的，可以分配主机里的任何文件或者目录给容器。
    2. Volumes: 由容器管理特定容器区域，不会影响目录下其他文件。
    3. tmpfs: 分配主机内存给容器
      (数据持久化文档：https://docs.docker.com/get-started/05_persisting_data/)

    镜像网络

    1. Host：直连到主机网络命名空间（NIC）
    2. Bridge：Docker管理自己的私有网络，通过NAT做内部网络和NIC的映射。

    为什么Docker的隔离性不能完全保证

    它提供了直连操作系统的方式，比如bind mount可以让操作系统的存储空间直接对容器可见，Host的网络连接方式会让容器直连NIC。
